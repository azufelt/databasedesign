Return & Report

Final Project Presentation

Return & Report was an idea that came to life by recognizing a real need amongst my own, real-world, experiences. As I serve and participate in local groups, boards and councils affiliated with The Church of Jesus Christ of Latter-day Saints, I realized that our group did not have a standard for project planning, tracking and various other leadership responsibilities.

As I've spent the last few years being a big supporter of the new Gospel Living (Circles) App, I thought it could use some new integrations. So, Return & Report is a project management application that is focused on integrating into the already existing Circles and other Church authenticated sites.

Our test case assumes that a user has already created an account, however the queries to create a user are still included, for representational purposes.

As you go through the inter-active UI you'll see that it is focused around a mobile application, however the screens have been widened to also include the appropriate CRUD statements.

Design Rationale:

The rationale for the design was based on the already exisiting UI elements, from within the production Circles App. Beyond that, I wanted to show that a single user can belong to multiple Circles Groups, and each group can have multiple project workspaces, and then within each project there can be several tasks. The tasks can be assigned to someone who becomes the task "owner". The tasks also integrate a status, which could be expounded upon in a production application, for now it is represented within the UI by a green/yellow/red circle. The status is a working table that is referenced when a user selects one of the circle status markers, then the task would use an 

UPDATE
to add the foreign table id as a reference to the project task.

Reflect:

Over-all I'm extremely happy with where I got with the project. I of course have so much more to learn about how powerful databases are, but I'm off to a really good start, and have an excellent foundational understanding of what it takes to design and build a database.

Some pieces that I learned and will take with me was the ability to decrease how much data I store in a table, but then join it with another table to actually produce data results or authentication that is needed. This is also a piece I struggled with, because I WANTED to be able to store everything that I needed in a view in one table so I could esily get to all of the things. But that doesn't allow for scalability, good maintenance pracices, as well as can really limit and even further complicate how statistics get drawn from the tables.

This database design project may be done, but I intend to flesh this out further and take it to production, either with the integrated circles use, or for some other various platforms.
